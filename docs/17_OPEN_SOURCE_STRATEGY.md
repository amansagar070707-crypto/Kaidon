# Open Source Strategy

## Purpose

Define how Kaidon should win as OSS.

## Core Strategy

- keep runtime protocol open
- keep plugin boundary open
- keep event schema documented
- ship strong local-first workflows

## What Should Be Open

- contracts
- runtime core
- SDK
- CLI
- local Studio
- template system

## What Can Stay Optional Later

- hosted control plane
- managed observability
- verified marketplace services

## Why This Works

Open protocols create trust. Managed convenience creates optional revenue.

## Tradeoffs

- Open core reduces lock-in leverage.
- It increases ecosystem adoption and contributor trust.

## Alternatives

- closed hosted product first
- source-available without protocol openness

## Verified Facts

- The project is explicitly positioned as open source.

## Assumptions

- The strongest adoption vector is developer trust plus interoperability.

## Speculative Ideas

- Kaidon can become the reference implementation of an open agent runtime
  protocol.
