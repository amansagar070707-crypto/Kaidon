# Roadmap

## Purpose

Provide a staged multi-release plan.

## Stage 0

- docs and research
- contract decisions
- architecture approval

## Stage 1

- monorepo cleanup
- shared package boundaries
- runtime protocol skeleton

## Stage 2

- durable runtime
- event taxonomy
- streaming transport

## Stage 3

- memory and context engine
- tool registry
- MCP connectors

## Stage 4

- SDK
- CLI
- template packs

## Stage 5

- Studio
- Cloud basics
- observability console

## Stage 6

- plugin system
- registry
- marketplace

## Tradeoffs

- Sequencing by substrate delays “wow” UI.
- It avoids building on sand.

## Verified Facts

- The directive explicitly defines future execution order and says not to start
  implementation yet.

## Assumptions

- Runtime and protocol maturity will decide later roadmap success.

## Alternatives

- feature-driven roadmap without substrate order

## Speculative Ideas

- A public roadmap can be split into OSS-core and hosted-commercial layers
  later.
