# Founder Notes

## Purpose

Capture the sharpest product guidance in one place.

## Notes

- Kaidon wins by making agent systems operable.
- Do not drift into “AI wrapper” positioning.
- Prefer open protocols over lock-in.
- Prefer reliable runtime truth over flashy UI demos.
- Use the ecosystem aggressively. Do not rebuild what is already good.
- Own the contracts that production teams actually need.

## Questions To Revisit Often

1. Why should a team adopt Kaidon instead of LangGraph plus custom glue?
2. What does Kaidon own that AI SDK should not own?
3. What does Kaidon reuse from Temporal instead of reimplementing?
4. What product surface proves the value fastest: CLI, Runtime, or Studio?
5. Which feature increases clarity, and which one only increases scope?

## Verified Facts

- The project directive explicitly prioritizes research, interoperability, and
  production-proven architecture.

## Assumptions

- The strongest long-term direction is “agent operating system,” not “best
  consumer AI app.”

## Tradeoffs

- This is a more ambitious category than agent framework tooling alone.
- It is also more strategically defensible if the seams are owned well.

## Alternatives

- IDE plugin first
- hosted app builder first
- closed platform first

## Speculative Ideas

- If Kaidon gets the protocol and runtime right, other products may build on it
  without Kaidon needing to own every user-facing workflow.
